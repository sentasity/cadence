Cadence is a Claude Code plugin that turns "I want to build X" into working, validated software — without asking you to read a single line of code to know it's right. This first lesson gives you the mental model, then gets the plugin installed.

## Why the mental model comes first

You are great at running operations and shaping ideas. You are not going to verify a change by reading a *diff* — the line-by-line list of what changed in the code. Cadence is built for exactly that situation, so you have to understand *what it promises* before the install steps will mean anything.

Here is the whole promise in one sentence: **work moves forward in six visible phases, each phase produces something you can read, and nothing advances to the next phase until you say so.**

<Aside type="note" title="Jargon, glossed once">
A few words you'll see throughout the course, in plain English:
- **Phase** — one named step of the pipeline (brainstorm, design, plan, execute, audit, validate).
- **Artifact** — the thing a phase hands you: a folder, a document, a commit you can open and read.
- **Gate** — the explicit pause between two phases where you approve before anything continues.
- **Drift** — when the work quietly stops matching what you agreed to. Cadence's whole job is to make drift impossible to do silently.
</Aside>

## The six-phase pipeline

Cadence runs your idea through six core phases, in order. Each one has a slash command and produces an artifact you can read:

1. **Brainstorm** (`/c-brainstorm`) — one question at a time turns a fuzzy idea into a short written stub.
2. **Design** (`/c-design`) — the stub becomes a full, plain-English design folder.
3. **Plan** (`/c-plan`) — the approved design becomes an exact, step-by-step plan written for the machine to follow.
4. **Execute** (`/c-execute`) — the plan becomes real code, reviewed twice before it lands.
5. **Audit** (`/c-audit`) — an independent check proves the code matches the plan.
6. **Validate** (`/c-validate`) — a post-deploy walkthrough confirms the live thing actually works, then marks the work done.

There are also three **diagnostic** commands you can run any time, off to the side of the pipeline: `/c-check`, `/c-find-bugs`, and `/c-explain`. That is nine commands in total. You meet the diagnostics in the last deep dive.

<Aside type="tip" title="The through-line">
You never have to trust the code by reading it. You trust the **gate** — the explicit, human-approved pause between every phase. That is what makes Cadence usable by someone who can't (or won't) read diffs.
</Aside>

## Install the plugin

The plugin installs once and then works in every repository on your machine.

<Tabs>
  <TabItem label="Install commands">
```text
/plugin marketplace add sentasity/cadence
/plugin install cadence@cadence
```
  </TabItem>
  <TabItem label="What you'll see">
After a session restart, all nine `/c-*` commands are available. There is no separate setup step — the first time you run `/c-brainstorm` in a fresh repo, Cadence offers to scaffold its config for you (you'll do that in the next lesson).
  </TabItem>
</Tabs>

<Aside type="caution" title="Restart your session">
The commands appear only after you restart your Claude Code session. If `/c-brainstorm` isn't recognized, restart and try again.
</Aside>

## Confirm the nine commands are there

  <Fragment slot="step-1-title">Restart your Claude Code session</Fragment>
  <Fragment slot="step-1-body">After `/plugin install cadence@cadence`, fully restart the session so the plugin's commands register.</Fragment>
  <Fragment slot="step-2-title">Type a slash and look for `/c-`</Fragment>
  <Fragment slot="step-2-body">Begin typing `/c-` in the prompt. You should see the six core commands (`/c-brainstorm`, `/c-design`, `/c-plan`, `/c-execute`, `/c-audit`, `/c-validate`) and the three diagnostics (`/c-check`, `/c-find-bugs`, `/c-explain`).</Fragment>
  <Fragment slot="step-3-title">You're ready</Fragment>
  <Fragment slot="step-3-body">Nine commands visible means the plugin is live in this repo and every other repo on your machine. Move on to your first brainstorm.</Fragment>

Throughout the quickstart you'll build one tiny feature — a function that prints the current time — and at the end compare your output against the finished worked example shipped with Cadence at `examples/hello-cadence/`. Same shape of artifacts means you did it right.
